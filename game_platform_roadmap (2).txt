REDIRECT TO PLATFORM GAME PAGE
================================================================================

When a user clicks a game linked to Steam, Epic, GOG etc:

FLOW:
    User clicks "View on Steam" button
        ↓
    Frontend calls your backend API
        GET /api/games/{id}/redirect?platform=steam
        ↓
    Backend looks up the URL for that game + platform
    Backend logs the click (analytics)
    Backend returns: { "url": "https://store.steampowered.com/app/..." }
        ↓
    Frontend receives the URL
    Frontend does: window.location.href = url
        ↓
    User lands on the platform page

WHY GO THROUGH BACKEND (not hardcode URL in frontend):
    - Track which platforms users click most (analytics)
    - Validate the URL still exists
    - Rate limit abuse
    - Change URLs without redeploying frontend
    - Log data for "best time to buy" features later

SEPARATION OF LOGIC
================================================================================

MENTAL MODEL:
    Browser (Frontend — HTML/CSS/JS)
        ↕  HTTP/JSON API calls ONLY
    Go Backend (Pure JSON API)
        ↕  SQL queries ONLY
    PostgreSQL / Redis (DB layer)

RULES (never break these):
    - Frontend NEVER touches the DB directly. Ever.
    - Backend NEVER renders HTML. Ever.
    - DB is NOT exposed outside Docker network. Ever.
    - Frontend and backend only talk through defined API contracts.

WHAT EACH LAYER IS RESPONSIBLE FOR:
    Frontend:
        - Rendering UI
        - Calling backend APIs
        - Handling user interactions
        - Redirecting users

    Backend:
        - Business logic
        - Auth validation
        - DB queries
        - Calling external APIs (RAWG, Steam, IGDB)
        - Returning JSON responses

    Database:
        - Storing and retrieving data
        - Nothing else
        - Never directly reachable from frontend or internet

CONTAINERIZATION (DOCKER)
================================================================================

THREE CONTAINERS, ONE NETWORK:

    docker-compose.yml
    │
    ├── frontend   (nginx serving static files)  → port 80 exposed to internet
    ├── backend    (Go app)                       → port 8080 internal only
    ├── postgres   (PostgreSQL DB)                → port 5432 internal only
    └── redis      (Redis cache)                  → port 6379 internal only
    │
    └── all on: gameplatform_network

PORT EXPOSURE RULES:
    - ONLY frontend port 80 is open to the outside world
    - Backend, PostgreSQL, Redis are internal only
    - No one on the internet can directly hit your DB or backend

HOW CONTAINERS TALK TO EACH OTHER:
    Inside Docker network, use service name (not localhost):
    - Backend connects to DB at:     postgres:5432
    - Backend connects to Redis at:  redis:6379
    - Frontend calls backend at:     http://backend:8080
    (Docker resolves these names automatically inside the network)

DOCKER COMPOSE STRUCTURE:
    services:
      frontend:
        ports: ["80:80"]          ← only this is public
        networks: [gameplatform_network]

      backend:
        expose: [8080]            ← internal only, NOT ports
        networks: [gameplatform_network]
        depends_on: [postgres, redis]

      postgres:
        expose: [5432]            ← internal only
        networks: [gameplatform_network]

      redis:
        expose: [6379]            ← internal only
        networks: [gameplatform_network]

    networks:
      gameplatform_network:

CI/CD — DETAILED BREAKDOWN
================================================================================

FULL ZERO DOWNTIME FLOW:

    You: git push origin main
            ↓
    GitHub Actions starts automatically
            ↓
    ┌─────────────────────────────┐
    │  Job 1: TEST (all branches) │
    │  - go test ./...            │
    │  - go vet ./...             │
    │  - go build                 │
    └─────────────────────────────┘
            ↓ only if tests pass
    ┌─────────────────────────────┐
    │  Job 2: BUILD (main only)   │
    │  - docker build             │
    │  - tag image with git SHA   │
    │  - push to AWS ECR          │
    └─────────────────────────────┘
            ↓
    ┌──────────────────────────────────────────┐
    │  Job 3: DEPLOY (zero downtime)           │
    │                                          │
    │  Old container: still serving traffic    │
    │  New container: starts up                │
    │  Health check: /health endpoint passes   │
    │  Traffic: ALB switches to new container  │
    │  Old container: drains, then stops       │
    └──────────────────────────────────────────┘
            ↓
    App is live. No downtime. No dropped requests.

ZERO DOWNTIME IS ACHIEVED BY:
    1. Graceful shutdown in Go
       - Go app listens for SIGTERM signal
       - Finishes all in-flight requests before shutting down
       - New container is healthy before old one dies

    2. Health check endpoint (/health)
       - Load balancer (ALB) pings this before routing traffic
       - Only sends traffic once new container responds healthy

    3. Watchtower (on EC2, simple setup)
       - Runs as a Docker container
       - Watches for new images in ECR
       - Automatically pulls and restarts your app

FILE THAT CONTROLS EVERYTHING:
    .github/workflows/deploy.yml
    (One file in your repo. GitHub reads it automatically.)

THREE JOBS IN THAT FILE:
    - test     → runs on every push to any branch
    - build    → runs only on push to main
    - deploy   → runs after build succeeds

AWS IAM — SECURITY DONE RIGHT
================================================================================

IAM = Identity and Access Management
It controls WHO can do WHAT on your AWS account.

THE WRONG WAY (never do this):
    - Use root AWS account for everything
    - Store AWS access keys in your code
    - Give everything full admin permissions

THE RIGHT WAY:

    Root AWS Account
        ↓ (only used once — to create IAM users. Never used again.)
        ↓
    ┌─────────────────────────────────────────────────────┐
    │ IAM User: "dev-user"                                │
    │ Used by: You (personal CLI access, local dev)       │
    │ Permissions: only what you need for development     │
    └─────────────────────────────────────────────────────┘
        ↓
    ┌─────────────────────────────────────────────────────┐
    │ IAM Role: "ec2-app-role"                            │
    │ Attached to: EC2 instance                           │
    │ Permissions: pull from ECR only. Nothing else.      │
    └─────────────────────────────────────────────────────┘
        ↓
    ┌─────────────────────────────────────────────────────┐
    │ IAM Role: "github-actions-role"                     │
    │ Used by: GitHub Actions pipeline                    │
    │ Permissions: push to ECR + SSH to EC2 only          │
    └─────────────────────────────────────────────────────┘

WHY SEPARATE ROLES:
    - If EC2 is compromised, attacker can only pull images — nothing else
    - If GitHub Actions is compromised, attacker can push images + SSH — that's it
    - Principle of Least Privilege: each identity gets minimum permissions needed
    - This is exactly what interviewers ask about

PRINCIPLE OF LEAST PRIVILEGE:
    Every user, role, and service gets ONLY the permissions it needs.
    Nothing more. This limits damage if anything is ever compromised.

CV BULLET THIS EARNS YOU:
    "Implemented AWS IAM with least-privilege roles for EC2, CI/CD pipeline,
     and developer access — separating concerns and limiting blast radius"
